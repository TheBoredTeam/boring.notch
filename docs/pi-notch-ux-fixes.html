01 — Content eats the header

The tall Pi panel squeezes the tab row to zero.

Screenshots 4 & 5, same session, same settings: the Home / Shelf / Pi switcher is present on Home (short content) and gone on Pi (tall, streaming content).

Not a tab-layout bug — the switcher is shared: TabSelectionView lives only in BoringHeader, rendered once for every tab, above the content. The cause is height. The Pi panel is sized by its own content — PiAgentView.reportDesiredHeight() → measuredContentHeight → openPanelHeight — and the answer is greedy (maxHeight: .infinity + .layoutPriority(1)). When the Pi content is tall (prompt + done + 4 chips + the Connect CTA + the answer, as in shot 4), it claims the entire panel height and the BoringHeader row above it compresses to nothing — so the switcher vanishes. On Home/Shelf the content is short and fixed (shot 5), so there’s always slack for the header. Pi just measures and grows as if the header row weren’t there.

shots 4 vs 5 · header present on short Home · gone on tall Pi — height, not width
Height math

A · Make the height math header-aware recommended

Reserve the tab row: openPanelHeight = headerHeight + contentHeight, and cap the answer’s max at panel − headerHeight. The switcher can never be squeezed out.

  • Exactly your instinct — Pi stops pretending the header isn’t there.
  • No new layout; the fixed-size tabs already assume this slot exists.
+ correct by construction
PiAgentView.reportDesiredHeight + BoringViewModel height clamp

B · Pin the header’s layout priority structural

Give BoringHeader a higher .layoutPriority than the Pi content so the answer yields the header’s row first when space is tight.

02 — Scroll indicator

The answer’s scroll pill is too heavy.

From screenshot 2 (“make this smaller”) and visible again in screenshot 3 beside the RSVP line: a fat, always-on vertical bar.

PiAgentView.answer sets .scrollIndicators(.visible), forcing the legacy overlay scroller to stay on at full weight even when there’s barely anything to scroll. Combined with the 28 pt comfort footer it reads as a permanent chrome rail down the panel’s right edge — exactly the “pill” in the crop. The fix is to stop pinning it visible; the affordance the panel actually needs (knowing more text exists) is already carried by growth + the comfort footer.

screenshot 2 · the lone vertical pill is the overlay scroller thumb
Indicator

A · .automatic recommended

Indicator fades in only while scrolling, then disappears — the native macOS overlay behavior. Thin, transient, present exactly when relevant.

+ one-token change, native feel
PiAgentView.swift · .scrollIndicators(.automatic)

B · .never cleanest

Hide it entirely. The panel already grows with content and the 28 pt footer signals “more below.” Zero chrome on the answer.

+ purest look · − no scroll-position cue at max height
PiAgentView.swift · .scrollIndicators(.never)

C · Thin inset custom rail most work

Replace the system scroller with a 3 pt rounded indicator inset from the chin, tinted to the toolkit accent.

+ on-brand · − custom overlay + scroll-fraction plumbing

03 — Corner geometry

The corners read jagged — quadratic, not squircle.

From screenshot 3: the open panel’s rounded chin still looks “squared and jagged” even after the 24 pt corner-fix.

NotchShape.path(in:) builds every corner with a single addQuadCurve. A quadratic arc isn’t the continuous (squircle) curvature macOS uses for its own notch and rounded rects — the curvature jumps at the tangent, so the eye reads a faint “corner” instead of a smooth flow, and the hard clip edge aliases against the bright aurora. Bumping the radius (the 24 pt fix) made it rounder but not smoother. The lasting fix is to match Apple’s curvature with a cubic.

screenshot 3 · chin corner shows a curvature kink + aliased clip edge
Corner

A · Continuous-curvature path recommended

Rebuild each corner with a cubic that holds curvature into the straight (≈ the .continuous superellipse). The chin flows instead of kinking.

  • Matches the hardware notch + every macOS rounded rect.
  • Geometry-only; animatableData stays the two radii.
NotchShape.swift · replace addQuadCurve with addCurve

B · Radius + 0.5 px hairline mask cheap

Keep the quad, raise the chin to ~26 pt and inset a sub-pixel mask so the clip edge doesn’t alias against the glow.

+ smallest diff · − still subtly non-continuous
ContentView.swift · openBottom + mask inset

C · .continuous RoundedRectangle where free partial

For surfaces not bound to the housing silhouette (the answer card, chips) use a continuous rounded rect directly. Doesn’t fix the notch chin itself.

− leaves the actual notch corner untouched
PiAgentView.swift · card backgrounds

04 — Glow intensity · display targeting

The aurora is too loud, and it’s on every screen.

From screenshot 3: “the gradients are too much distraction / big,” and the notch should “render on one screen if two screens are present.”

Two independent knobs. Intensity: the aurora blooms at radius 0.30–0.70·w and core opacity up to 0.92–1.0 — tuned to show off a brand palette, which reads as a distraction during ordinary work. Targeting: Defaults[.showOnAllDisplays] defaults to true, so adjustWindowPosition spins up a notch window per NSScreen. Upstream already supports the off-path — when false it renders only on the preferred screen. Both are choices, not new machinery.

grounded · upstream boring.notch + DynamicNotchKit (Nia) — single-display is the off-branch of the same switch
Glow

A · Dial intensity −40% quiet

Keep the aurora’s identity, just turn it down: core opacity ×0.6, radius ×0.8, desaturate one step. Present but no longer shouting.

+ keeps brand color · − still a colored field
PiPeekView.swift · aurora core/fade/opacity consts

B · Underglow only recommended

Drop the wing blooms; keep a thin brand-tinted glow hugging just under the chin (the underglow doc’s direction). The notch is lit, the screen isn’t washed.

  • Lowest distraction; still signals “Pi is working.”
  • Re-uses the inner-fade mask already in aurora.
PiPeekView.swift · collapse blooms → single chin band

C · Activity-tied bloom dynamic

Near-invisible while idle/thinking; a brief bloom on tool_start that settles back to underglow. Loud only at the moment that earns it.

+ expressive · − most motion to tune

Display targeting — pick the resting rule

One screen · preferred recommended

Default showOnAllDisplays = false. The notch lives on the preferred display only — the off-branch that already exists. Matches the ask directly.

Constants.swift · default false · already wired

Follow the active display option

A single notch that hops to whichever screen has the cursor / key window. One notch, always where you are — more code than flipping the default.
